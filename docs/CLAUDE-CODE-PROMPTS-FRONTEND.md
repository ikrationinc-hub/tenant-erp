## FE-3 — `<SchemaForm/>` ⚠️ THE ONE THAT MATTERS

```
Read frontend rules 1, 3, 6 in CLAUDE.md and §4 of docs/spec/Purchase-V2.md before starting.

This is the core bet of the whole project. Every form in this ERP renders through it. If a screen ever hardcodes a field, this component has failed and weeks 5-11 of backend work are wasted.

1. src/core/schema-form/field-types/ — ONE component per field type from the spec. Exactly these 13, no more:
   AutoGenerated (read-only) · Textbox · TextArea · Dropdown (master-backed, async options) ·
   Lookup (searchable, links another module) · DatePicker · Decimal · Currency · Percentage ·
   Toggle · Calculated (read-only, server-computed) · FileUpload · MultiUpload
   Registry keyed by field type. NEVER a switch on field type outside this registry (rule 6).

2. src/core/schema-form/compile-validator.ts — field-definitions → Zod schema at runtime, memoized by (module, entity, version). Mirrors what the backend does with AJV, so client and server validation cannot drift.

3. src/core/schema-form/SchemaForm.tsx:
   - Props: { module, entity, initialValues?, onSubmit, mode: 'create'|'edit'|'view' }
   - Fetches GET /field-definitions/:module/:entity via TanStack Query
   - Renders sections and fields from the schema — order, labels, mandatory, visibility ALL from metadata
   - React Hook Form + zodResolver from the compiled validator
   - Fields the schema marks non-editable render read-only. Fields it omits do not render at all. The frontend does NOT decide this (rule 4) — the backend already intersected definitions with permissions.

4. MONEY — rule 3. Currency/Decimal/Percentage receive and emit STRINGS. Never parseFloat. Never native arithmetic. Calculated fields display what the server sent; they are never computed client-side.

5. Section support: the schema groups fields into sections (Purchase Header, Shipment Details, ...). Render as AntD Card or Collapse per section.

6. Dynamic behaviour from metadata: visibility_condition_json (show/hide on another field's value), options_source (async master fetch, cached), cascading dropdowns (Country → City, per the spec).

7. Storybook-free — write a dev route /_dev/schema-form rendering a hand-written schema fixture covering all 13 types, so the renderer is testable before Purchase exists.

Tests:
- Each field type renders and round-trips a value
- Label comes ONLY from metadata — changing the fixture's label changes the DOM
- Mandatory from metadata → validation fires
- A field omitted from the schema does not render
- A non-editable field renders read-only and cannot be submitted
- Currency fields never produce a float — assert the submitted payload is a string
- Cascading dropdown filters correctly

Acceptance:
- /_dev/schema-form renders all 13 types from a fixture
- `grep -rn "label=\"" src/modules/` returns NOTHING
- No `any`
```

**Do not proceed to FE-4 until the grep is clean.** One hardcoded label is a broken window; the rest follow within a fortnight.

## FE-4 — `<SchemaTable/>` + navigation

```
=== SCHEMA TABLE ===
1. src/core/schema-table/ — generic grid over AntD Table:
   - Props: { module, entity, columns?, filters?, actions? }
   - Column definitions from field-definitions metadata (respects visibility + labels), overridable per screen for list-specific columns
   - Server-side pagination, sorting, filtering (backend rule 10 — never fetch-all-and-filter-client-side)
   - URL-synced state so a filtered view is shareable and survives refresh
   - Row actions gated by permission
   - Loading skeletons, empty state, error state
2. Column widths, resize, sticky header, dense mode. Traders live in these grids all day.

=== NAVIGATION ===
3. src/core/navigation/ — renders GET /menus into an AntD Menu. Tree, icons, sort_order, parent_id nesting. NO hardcoded route array (rule 2).
4. Dynamic React Router routes registered from the menu tree. A path not in the user's menu → 404, not a blank screen.
5. Breadcrumbs derived from the same tree.
6. Menu refetches on company switch and on role change.

Tests: table pagination/sort/filter hit the server not the client; URL state round-trips; nav renders from a mocked /menus fixture; a menu item the user lacks permission for is absent; unknown route → 404.

Acceptance:
- `grep -rn "path: '/" src/app/router` shows no business routes — only the shell + auth
```
